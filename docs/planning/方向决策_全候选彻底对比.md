## 0. 一条主线（先看这个）

**保底 CCF-C 的最大变量不是"方向多新"，而是"有没有公开数据 + 公开 baseline + 能不能跑出真实结果"。**
越新的概念公开数据越少 → 越难保底。所以策略是：
> **把你的核心发明定义成一个"方法"——per-run 随机 canary/honeytoken 污点标记 + 运行时取证——
> 它生态无关，可在多个面复用。** 用"有公开数据的面"保底，用"新面"拉高上限。

## 2. 逐项体检

### A｜技能能力一致性核验 + 蜜罐取证 + 阻断（你现在的方向）
- **拥挤**：Agent Skills 检测/基准已饱和（USENIX'26 2602.06547、MalSkillBench 2606.07131、BIV 2605.11770）。
  但"运行时观测的声明-实际一致性 + 蜜罐外传取证 + 阻断"这条别人没合起来做（BIV 是静态预测、只核验）。
- **数据/baseline**：🟢 全现成（本地已拉取 3,945 恶意 + 4,001 良性 + 一整套 baseline）。
- **复现**：🟢 最低。**保底把握最高。** 上限一般（顶会嫌 incremental）。
- **结论**：**保底首选**，最省时间，最稳妥。

### B｜跨 App 共享上下文投毒——检测/防御/基准（ChatGPT Apps，最推荐的"新且能做"）
- **拥挤**：🟢 仅 1 篇**攻击**论文（Confused ChatGPT 2606.00485，Jha/Lin，2026-05），**明确没做检测/防御/基准**。
- **机会**：**做"第一个跨 App 上下文投毒的检测方法 + 防御 + 评测基准"本身就是贡献。**
  你的 canary 思路直接迁移：给每个 App 写入共享上下文的内容打**来源污点标记**，检测"另一个 App
  是否读取/据此越权"→ 跨 App 污点追踪。
- **复现**：🟡 需 ChatGPT Apps SDK 访问 + 自建评测（可用云算力跑多模型）；工作量比 A 大，但可控。
- **结论**：**新颖度高 + 仍可复现 + 蹭你的老本行**，是"保底与上限的最佳平衡"。

### C｜AI Control / Agent-as-Insider-Threat 监控协议（DeepMind 2026-06，最高上限）
- **拥挤**：🟡 很热、大厂主导（Redwood/DeepMind/UK AISI）。硬拼整框架赢不了。
- **好消息**：**有开源测试台 ControlArena**（UK AISI + Redwood 的 Python 库，含 monitors / 微协议 /
  可信模型编辑不可信输出），**用云模型就能跑控制实验**——保底比想象中可行。
- **机会（缩小到你能赢的点）**：提出**"欺骗式监控器"**——把 honeytoken/canary 作为 control 协议里的
  一个低成本 monitor，抓不可信 Agent 的外传/越权，在 ControlArena 上对比现有 monitor 的
  suspicion score / 安全-可用性权衡。
- **复现**：🟡 中（威胁模型是"模型本身是坏人"，比"技能是坏人"更难，reviewer 要求高）。
- **结论**：**上限最高（可能冲击 > CCF-C）**，但要把范围收得很窄，风险中等。

### D｜异步自治云 Agent（Cowork/Codex Cloud）安全
- 🟢🟢 最新最不卷，但 🔴 闭源、无公开数据、要自建开源替身运行时。**保底最弱**，只作高风险可选赌注。

### E｜Assistant Axis / 人格漂移
- 🟢 较新，但属机制可解释性/激活操控，**与你系统安全技术栈不符=转行**。除非你真的想学 interp，否则不选。

### F｜Agentic 浏览器 / 电脑操作 Agent 安全
- 🔴 媒体和学界都在猛打（"Atlas/Comet 的 prompt injection 无法根治"），**很挤、难差异化**。不建议。

### G｜GRAM / 两用知识模块
- 需**从头预训练**大模型，纯大厂玩法，**你做不了。放弃。**

## 3. 最终推荐（作为你的架构师/项目经理）

**首选组合（保底 + 上限兼顾，且都用你的 canary 方法当主线）：**
> **主线 = "canary 污点 + 运行时取证"方法**，落两个面：
> - **面①（保底）= A**：技能凭据外传的一致性核验（公开数据现成，稳拿实验结果）；
> - **面②（上限）= B**：跨 App 共享上下文投毒的检测/防御（新、防御空白，做出第一个基准）。
>
> 论文叙事：**"一个生态无关的 canary 污点运行时取证方法，既能查技能供应链外传、又能查跨 App
> 上下文投毒"**——方法通用性 = 新颖度，公开数据 = 保底。

**若想冲更高（愿冒中等风险、时间充裕）**：把面②换成 **C（AI Control 里的欺骗式监控器，跑 ControlArena）**，
上限可能突破 CCF-C，但要接受 reviewer 门槛更高。

**不建议**：D（保底太弱）、E（转行）、F（太挤）、G（做不了）。

**关于云算力**：A/B/C 都能用云 API 跑多模型对比（统一 prompt、temperature=0、固定版本、报成本延迟）；
但**恶意样本默认不上传闭源云**——敏感样本用云上你可控的开源模型实例（vLLM）跑，闭源仅跑脱敏/合成子集。
