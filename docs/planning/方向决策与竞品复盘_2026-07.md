## 1. 为什么"拥挤/思路接近"不等于发不出来

- 会被拒：**纯重复、无清晰增量**；或**吹嘘并不存在的新颖性**。
- 仍能发（尤其 CCF-C / SCI 中低区）：有一个**诚实、清楚的 delta**——新测量维度 / 揭示问题的独立复现 / "已知攻击但没人做防御"的缺口 / 资源受限下可部署的变体——再加**扎实 + 可复现 + 引全 + 定位准**。
- 拥挤领域照样天天发论文；关键是**别装头牌，把 delta 讲清楚**。

## 3. 候选落地路线（含诚实定级）

### 甲｜独立复现 + 公平对比 + 边界测量（复现/测量型）
- **拥挤度中偏高**：MalSkillBench 自带 baselines；强方法各自做过对比；不能卖"首个对比"，只能卖"独立第三方复现 + 复现难点 + 统一底座边界/负向测量"。
- **可行性**：数据集 + MalSkillBench baselines + RSA 代码公开；有云 API 后 LLM-在环方法可较完整复现。
- **定级**：workshop / reproducibility track / 中文核心 / SCI Q3–Q4。**价值真实但天花板低。**

### 丙｜给 cross-app 上下文投毒做"可复现防御/检测"（**当前推荐**）
- **delta 清晰**：攻击已发表(2606.00485)、可信可引，但**防御侧只有临时启发式**——做一个**上下文来源标记(provenance tagging) + 来源隔离/授权对齐**的确定性检测/缓解，在**开源多 App/多智能体模拟 harness**上评测（规避闭源 ChatGPT Apps 运行时，可复现）。
- **复用已有资产**：零误报确定性决策、声明↔实际一致性、canary/取证工程。
- **可行性（有云 API 后成立）**：用 DeepSeek/Kimi/Claude/Gemini 做 harness 里的 agent；对比"通用 IPI 防御（spotlighting / dual-graph provenance 2605.26497 / CaMeL）迁移到持久跨 App 场景"的效果差。
- **风险**：IPI 防御是红海，必须把"持久、跨 App、first-party API 写入"这一**特定场景**的 delta 讲死，否则会被当通用 IPI 防御的子集。
- **定级**：若执行扎实 + 诚实基准，**CCF-C / SCI 中低区可冲**。

### 乙｜彻底换更安静的邻近题目
- 保留选项：若丙验证下来仍太挤，可让 AI 再重扫一个更冷的题（代价是丢掉现有 skill 安全积累）。

1. **推荐主攻：丙**（cross-app 上下文投毒的可复现防御），甲作为兜底/补充章节（复现+边界测量可并入丙的实验）。
