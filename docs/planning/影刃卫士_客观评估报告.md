## 三、代码质量点评（供优化）

**优点**：结构清晰、注释到位（团队零基础背景下这点很务实）、引擎可插拔、失败降级（LLM 挂了不拖垮静态）、测试意识好。作为大创工程完成度，这部分是合格甚至偏上的。

**需要改的（按优先级）**：
1. **评分是一堆手工魔法数字的加法**（`pipeline.py:_score_static_findings`：+0.50/+0.20/+0.30…）。这在工程上能跑，在论文里会被直接质疑"阈值怎么来的、是不是照着样本调的"。→ 要么改成可学习/可校准的模型，要么至少做**消融实验 + 阈值敏感性分析**，并说明阈值是在独立验证集上定的。
2. **SSD 四子任务号称"并行"，实际是顺序调用**（`llm_judge.py:546-549`，代码里自己留了 `TODO: asyncio`）。论文如果写"并行降低延迟"就与实现不符。要么真做异步并行，要么改措辞。
3. **JSON 解析靠正则兜底**（`_parse_json_response`），小模型输出不稳时假阴性风险高。建议上 structured output / function calling / 重试校验。
4. **AST 隐写、HTML 注释注入**等你 `research.md` 自己列的盲区，解析器目前不覆盖。
5. **可复现性**：论文要开源代码 + 固定随机种子 + 明确模型版本/温度，现在温度 0.3 + 小模型，跨次结果会抖，需要报告方差而非单点数。

## 五、发论文可行性（认真、分档说）

### 先说残酷的事实
- **CCF-A/B 安全顶会（S&P/CCS/USENIX Security/NDSS）、SCI Q1**：以三人、大模型零基础、8 个月周期，**不现实**。这些要求方法新颖性 + 大规模严谨评测 + 极高写作/审稿博弈能力，和"陈同学"的差距不是靠这一个项目能补的，别拿这个标准折磨自己。
- 你 `PROJECT_CONTEXT.md` 自己定的目标是 **SCI Q2 / CCF-C~B**。**把 B 划掉，锁定 C / 中文核心 / SCI Q3–Q4**，这才是能够到的。

### 现实可行的投稿路径（按性价比排序）
1. **中文核心期刊**（推荐优先）：如《网络与信息安全学报》《信息安全学报》《计算机应用》《计算机工程》等。选题新、有真实攻击背景、有可用系统 + 一定规模实验即可，对英文写作要求低，最适合你们起步。
2. **CCF-C 会议 / workshop**：门槛比顶会低很多，接受"系统+评测"型工作。
3. **SCI Q3–Q4 期刊**：需要英文写作，但选题和这个方向匹配的开放获取期刊不少。
4. **软著 + 大创结题**：这个几乎是稳的，代码量和完成度足够。

### 想真正发出去，必须补齐的 4 件事（缺一不可）
1. **建出差异化模块**：动态沙箱 + 蜜罐 + 时间伪造触发延迟激活，这是你唯一能写"新"的东西。哪怕只做一个做深（比如"蜜罐凭据 + 沙箱行为监控检测 Agent 技能数据窃密"），也比五个都浅强。
2. **数据规模上量级**：15 个样本→**至少数百个恶意 + 数百个正常**。优先用你 research 里列的公开基准（若真实存在，见下条警告），并明确划分训练/验证/测试。
3. **对齐 baseline + 消融**：拿公开的同类工具作对照，做消融证明"动态引擎/蜜罐分别贡献了多少检出提升"。没有对照的检出率数字在论文里没有说服力。
4. **逐条核实所有引用**（见下节，这是红线）。

### 收窄叙事
论文不要再写"5 大创新点"。审稿人对"创新点清单"天然不信任。**挑一个能自圆其说、且实验能支撑的核心贡献**，其余作为工程支撑或 future work。

## 六、两个必须立刻处理的硬伤

### 🔴 硬伤 1：引用/竞品资料可能是 AI 幻觉
`research.md`、`PROJECT_CONTEXT.md` 里有大量**非常具体**的东西：arXiv 编号（如 `2604.06550`、`2604.22888`）、精确 F1（0.8834、96.95%、93%）、工具名与 GitHub 地址、"某某大学/某某所"归属。这些是 AI 助手联网整理的，而**AI 编造以假乱真的文献/数字是高发问题**。

- 论文里引用**一条不存在的文献 = 审稿人当场失去信任 = 拒稿**，性质等同学术不端。

- **凡是查不到原始出处的，一律不许进论文，也不要在申报书里当依据。**
