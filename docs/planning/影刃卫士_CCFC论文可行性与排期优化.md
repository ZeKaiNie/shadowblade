## 一、底层原理复述（先说清楚我们到底在赌什么）

项目的技术内核可以浓缩成一句话：

> **不靠"猜"，而靠"抓现行"**——把陌生技能放进布满蜜罐诱饵的隔离沙箱里真实执行，
> 用运行时行为监控捕获它对假凭据/身份文件的窃取与外传，蜜罐的**每次随机标记**在
> 网络/子进程参数里现形即构成确凿证据。

这套"动态沙箱 + 主动欺骗（deception）+ 行为取证"的组合，是本项目区别于
"静态扫描 + LLM 语义研判"这类**已经很拥挤**的方向的关键。静态+LLM 那半边，
坦率讲创新性已经不够撑一篇 CCF-C；能立住的是动态+蜜罐这半边。

## 二、完整开发后，能不能发 CCF-C？

### 结论

**有可能，但不是靠"把功能做完"就能到——决定成败的是实证严谨度（数据+对照+消融），
不是代码量。** 把话说死一点：

- 只把 6 个引擎都写完、指标停留在"15 个样本 100% 检出" → **发不了**，会被当作
  demo/工程报告，审稿人一句"评测不足、无法证明泛化"就退。
- 若补齐下面"硬门槛"清单 → **CCF-C 会议 / SCI Q3–Q2 期刊是现实可及的**（如
  AsiaCCS、ACNS、DIMVA、SecureComm 这类 CCF-C 安全会，或 Computers & Security、
  Journal of Information Security and Applications 这类期刊；中文核心如
  《网络与信息安全学报》《信息安全学报》更稳）。
- CCF-A/B 顶级（S&P/CCS/USENIX Sec/NDSS）**仍然不现实**，别用它折磨自己。

### CCF-C 的硬门槛清单（缺一个都危险）

1. **数据规模与真实性**：样本从 15 → **数百级**（恶意 + 良性都要）。恶意样本要么
   来自真实 ClawHub 事件（最有说服力），要么按**明确攻击分类学**（凭据窃取 / C2 外联 /
   身份文件后门 / 延迟激活 / 供应链投毒 …）系统构造，并**人工标注**。良性样本要覆盖
   正常技能的多样行为（会读文件、会联网的正常技能），否则误报率没意义。
2. **Baseline 对照**：至少和"纯静态""纯 LLM""已有开源工具"比。没有对照 = 没有论文。
3. **消融实验**：证明"动态引擎 / 蜜罐 / 时间伪造"各自带来多少增量检出，否则无法主张贡献。
4. **评测指标**：Precision / Recall / F1 / FPR，在**留出集**上报告；对良性样本单独做
   误报分析。当前"0% 误报"在 5 个良性样本上毫无统计意义。
5. **威胁模型 & 抗规避评估**：明确假设攻击者知道有沙箱，评估它能否**检测并规避沙箱**
   （反调试、环境探测、延迟激活）。这正是 libfaketime / 多次执行要解决的，也是论文加分项。
6. **novelty 论证**：用一节讲清楚和 2025–2026 已出现的 agent 安全工作的差异——把
   **"per-run 随机蜜罐标记 → ground-truth 外传取证"**和**"Agent 身份文件（SOUL/MEMORY）
   持久化后门检测"**作为拿得出手的差异点。

### 风险提示

- **赛道正在变拥挤**：AI Agent / MCP / 技能供应链安全 2025 年起论文和工具明显变多，
  投稿时"我们第一个做"的话术很危险。必须靠实证 + 具体差异点，而不是"新概念"取胜。
- **可复现性**：审稿人会要数据集/代码。提前把评测流水线做成一键复现，是隐形加分。

## 三、结合前沿，项目排期需要怎么优化

### 最重要的一条：现在就该动"数据 + 评测"，而不是继续堆功能

论文的生死线在评测。建议把优先级从"再写 mitmproxy/时间伪造/FastAPI/Dashboard"
调整为：**先搭数据集 + 评测框架，边补关键引擎能力边跑实验**。功能再多，没有数据一样发不出。

### 技术层面：一个必须正视的短板 —— 监控只到 Python 层

当前行为监控用的是 CPython `sys.addaudithook`，它**只能看到 Python 解释器层面的行为**。
一旦恶意代码用 `ctypes` 直接调 libc、跑非 Python 子进程、或用原生二进制，就**看不全**，
审稿人会直接质疑"沙箱可被绕过"。前沿做法是**内核级系统调用监控**：

- **eBPF**（如 Falco / Tracee / Sysdig）或 **seccomp + ptrace / strace** 在 syscall 层抓
  文件/网络/进程行为，覆盖非 Python 载荷，抗规避性强得多。
- 建议：动态引擎 v0.2 增加"内核级监控"后端，与现有审计钩子互补（钩子拿语义、eBPF 兜底）。
  这既是工程健壮性提升，也是论文里能写的一个真贡献点。

### 建议的分阶段排期（可直接替换 PROJECT_CONTEXT.md 里的规划）

**P0（论文地基，最高优先，1–2 月）**
- 攻击分类学 + 数据集：数百级样本，恶意按类别构造/收集 + 人工标注，良性覆盖多样行为。
- 评测框架：一键跑全量、输出 P/R/F1/FPR、支持 baseline 与消融开关。
- （已完成）动态沙箱最小闭环 → 作为被评测对象。

**P1（增强动态引擎的研究价值，2–3 月）**
- 内核级监控后端（eBPF/seccomp），补 Python 审计钩子的盲区（抗规避）。
- libfaketime 时间伪造 + 多次执行行为对比（抓延迟激活）——正好对应"抗规避评估"。
- 蜜罐机制形式化：标记设计、误报边界、外传取证的可靠性论证。

**P2（成文与投稿，1–2 月）**
- 跑齐 baseline + 消融，画图表；写威胁模型、novelty、局限。
- 先投一个 **CCF-C 会议或中文核心**试水；被拒拿审稿意见再迭代。

**P3（工程收尾，不占论文关键路径，穿插做）**
- FastAPI + Streamlit 做成 demo（论文/答辩/大创展示用），mitmproxy 流量解密作为可选增强。
- 这些**不要**放在论文关键路径上——它们不决定能否中稿。

### 可以考虑纳入的前沿角度（用于拔高 novelty，需自行核实具体文献）

- **供应链 provenance / 签名**：技能来源可信度、SBOM、sigstore 式签名，呼应"供应链"定位。
- **间接提示词注入（indirect prompt injection）**：agent 安全热点，但已很多人做，
  建议作为静态侧的一个子检测，别当主创新。
- **Agent 记忆/身份污染**：SOUL.md/MEMORY.md 持久化后门是个较新的具体切口，值得做深。
- **对已有 agent 安全 benchmark 的对接**：若有公认基准，接上去比自造数据更有说服力
  （具体基准名称请务必核实后再引用）。

## 四、一句话总结

底层原理（动态沙箱 + 蜜罐取证）**够格冲 CCF-C**，但门票是**数据集 + 对照 + 消融 +
抗规避评估**，不是功能数量。技术上最该补的是**把监控从 Python 层下沉到内核层（eBPF）**。
排期上把"数据与评测"提到最前，"FastAPI/Dashboard/mitmproxy"往后放。CCF-A/B 顶会短期
不现实，CCF-C / SCI Q3–Q2 / 中文核心是可以够到的现实目标。
