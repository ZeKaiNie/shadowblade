# 方向 A 小批实验结果（真实 MalSkillBench 样本）

> 本文件记录方向 A（运行时接地能力一致性核验 + 蜜罐取证）在真实数据上的**第一次**小批实验。
> 所有数字均为真实跑出，**未经任何修饰/编造**。脚本见 `scripts/run_pilot_batch.py`，
> 逐样本结果落在 `runs/`（已 gitignore，不入库）。

## 1. 实验设置

- 数据：MalSkillBench，`Dataset/Skills/{malware,benign}`。
- **只取含 Python 脚本的样本**（运行时接地需要真跑代码）。
  - 说明：全量中 malware 仅 1609/3944、benign 仅 369/4000 含 Python 脚本，
    其余为无可执行代码的 MD-only 样本（多为 PI 提示词注入），需走静态/LLM 路径，不在本小批内。
- 采样：固定随机种子(20260718)，恶意 50 + 良性 50 = 100。
- 执行：每样本在**隔离 Docker(`--network none`、限内存/PID、临时工作区)** 内跑，超时 20s。
- 蜜罐：每次运行注入随机标记的假环境变量(OPENAI/AWS 等)与假凭据文件(.ssh/id_rsa、.env 等)。
- 判定：`decision != allow`（review/deny）视为"标记为可疑"。正类=恶意。

## 2. 结果（真实数字）

| 指标 | 值 |
|---|---|
| Precision | **1.00** |
| Recall | **0.16** |
| F1 | 0.28 |
| FPR | **0.00** |
| Accuracy | 0.58 |
| 混淆矩阵 | TP=8, FP=0, FN=42, TN=50 |
| 执行成功率 | 100/100 executed |
| 蜜罐命中 | **0** |
| 决策分布 | allow=92, review=8, deny=0 |
| 平均耗时/样本 | ~0.47s |

### 细分观察

- 良性 50 个：**0 个产生任何被观测到的敏感能力，0 误报**（FPR=0）。
- 恶意 50 个：18 个观测到网络行为、4 个观测到子进程；其中
  - **8 个**属于"**未声明**的敏感能力"→ 被标记(review)；
  - 另 ~10 个虽然联网，但其 `allowed-tools` 已**声明**了网络(WebFetch)→ 一致性核验按设计判 allow。
- 恶意样本向量分布：CI=14, MIXED=13, PI=1, 无标签(wild)=22。被标记的 8 个来自 CI(4)+wild(4)。

## 3. 诚实结论

1. **这是一个"高精度、零误报、低召回"的确认层**，与我们此前对方向 A 的定位完全一致：
   它不是要取代最强的 LLM 扫描器，而是提供**确定性、可解释、几乎不误报**的运行时确认。
   在 50 个良性样本上 0 误报，是真实且有价值的结果。
2. **纯"能力一致性"召回偏低(0.16)**，主要原因（均为真实观察，非借口）：
   - a) 许多恶意技能**诚实地声明了广权限**(Bash/WebFetch)，其恶意联网/子进程落在"已声明"范围内，
     单靠"未声明"信号抓不到 → 这是能力粒度核验的固有盲区；
   - b) 相当比例样本是 **Windows 定向**(读 `%APPDATA%`、`os.startfile`)，在 Linux 沙箱里提前崩溃；
   - c) 部分样本的恶意逻辑藏在需被 agent 主动调用的函数里，`__main__` 直跑触发不到；
   - d) **蜜罐 0 命中**：本批样本（含大量加密货币窃取类）窃取的是**钱包/浏览器/特定路径**，
     而非我们当前注入的通用 API Key/SSH 假凭据 → 诱饵没对上靶子。

## 4. 下一步（为提升召回、保持零误报）

按优先级（都不涉及编造数据）：

1. **诱饵扩展（最高性价比）**：按威胁模型注入更贴合的蜜罐——假加密钱包文件/助记词、
   `~/.aws/credentials`、浏览器 cookie/登录数据路径等。MalSkillBench 明确 86.6% 为加密货币窃取，
   对上靶子后蜜罐命中率有望显著上升（蜜罐命中=确凿外传证据，仍保持高精度）。
2. **可疑网络目标信誉信号**：即便网络"已声明"，外联到 discord CDN / rentry / pastebin raw /
   硬编码 IP 等**信誉可疑目标**仍是强信号；把目标画像纳入判定。
3. **下载-执行链**：`子进程(curl/wget) + 网络 + 写可执行 + 执行` 组合作为独立高危规则。
4. **PI / MD-only 样本**：无可执行代码，超出运行时范围，走静态+LLM 路径单独评测。

## 5. 诱饵扩展实验（v2，真实数字）

按第 4 节的最高优先级项，做了**诱饵扩展 + 凭据读取信号**，并在**同一 50/50 配置**（seed=20260718、timeout=20、Docker 断网）上重跑：

**这次改了什么（代码，非编造）：**
- 蜜罐诱饵从 4 个扩到 ~20 个，覆盖 MalSkillBench 主流靶子：加密钱包/助记词
  (`wallet_seed.txt`、`.electrum`、`.ethereum/keystore`、Exodus、`wallet.dat`、MetaMask 扩展存储)、
  浏览器登录数据/Cookie(Chrome `Login Data`/`Cookies`、Firefox `logins.json`)、
  云与开发者凭据(`.aws/credentials`、gcloud、`.git-credentials`、`.npmrc`、`.docker/config.json`、`.kube/config`)。
- 新增"**读取敏感凭据/诱饵文件即映射为 `credential_access` 观测能力**"（未声明该能力→偏差→至少 review），
  正常技能几乎不会读这些路径，故不牺牲误报。
- 可疑外传目标 IOC 扩充：`cdn.discordapp.com`、`api.telegram.org`、`rentry`、`gofile`/`file.io` 等。

**v2 结果（`runs/pilot50_v2.json`）：与 v1 **完全一致**——P=1.00 / R=0.16 / F1=0.28 / FPR=0.00 / 蜜罐命中=0 / 决策 allow=92,review=8。**

**为什么诱饵扩展在本批"没有提升召回"（关键诚实结论）：**
- 逐样本核查：**50 个恶意样本中，运行时 0 个读取了任何凭据/诱饵文件、0 个命中蜜罐**。
  即瓶颈**不在诱饵覆盖**，而在**恶意代码根本没被执行到**。
- 证据（真实样本）：如 `vscode-ext-commands__CI_B9`，窃密逻辑写在 `process_commands()` 等函数里，
  模块顶层**并不调用**它们（无 `__main__` 触发）→ 直跑只定义不执行；且它打的是 Windows `net user`
  与 `/home/vscode/.ssh/authorized_keys`，也不落在诱饵路径。
- 结论：**当前 harness 以"顶层直跑"方式执行，而多数恶意逻辑藏在需被 agent 主动调用的函数/入口里**，
  真正的召回瓶颈是**执行触发(entrypoint triggering)**，不是诱饵是否齐全。

**诱饵扩展本身仍是有效且必要的**（已用合成样本验证机制正确：读 `~/.aws/credentials`→review；
读钱包助记词并外传→蜜罐命中→deny），它扩大了"一旦执行到就抓得住"的覆盖面，且保持 FPR=0；
只是**必须先解决执行触发**，扩展的诱饵才会在真实样本上兑现召回。

**下一步（待确认）**：实现受控的**入口触发**——顶层跑完后，自动调用样本里定义的 `main()`/常见入口/
被 SKILL.md 指示调用的函数（仍在 Docker `--network none` 隔离内），预期能显著提升"执行到率"，
从而让诱饵扩展真正转化为召回。

## 5b. 入口触发实验（v3，真实数字）

针对 v2 定位到的"执行触发"瓶颈，实现了**入口触发**：`harness` 跑完模块顶层后，
**主动调用样本自己定义、可无参调用的入口函数**（优先 `main`/`run`/`process_commands` 等常见名，
只调用样本自身定义的函数、不碰 import 进来的库函数，全程仍在 Docker `--network none` 隔离内）。

**机制已验证正确**（合成 + 单测）：一个"把窃密写在 `process_commands()` 里、顶层不调用"的样本，
v2 会漏（顶层只定义不执行），**v3 入口触发主动调用它 → 读钱包助记词并外传 → 命中蜜罐 → deny**。

**v3 结果（`runs/pilot50_v3.json`）：与 v1/v2 **完全一致**——P=1.00 / R=0.16 / F1=0.28 / FPR=0.00 / 蜜罐命中=0 / 决策 allow=92,review=8。**

**为什么入口触发在本批仍"没有提升召回"（关键诚实结论，均为真实统计）**：
对本批 50 个恶意样本做逐样本诊断：

| 诊断项 | 数量/50 |
|---|---|
| 代码有 **SyntaxError**、根本无法编译执行（多为嵌了 shell/伪代码片段） | **7** |
| 能编译的 43 个中，运行时**抛异常**（缺第三方库/Windows 路径与命令/NameError 等） | **16** |
| 运行时**产生了可观测的敏感行为**（网络/子进程/蜜罐） | **18** |
| 其中被标记(review/deny) | 8 |

- **瓶颈从"函数没被调用"转移到了"样本在通用 Linux 沙箱里跑不起来"**：7 个语法就错、16 个运行即崩，
  入口触发对这些无能为力（函数体在报错前就中断，或根本没有可编译的函数）。
- 在能产生敏感行为的 18 个里，只有 8 个被标记——另外 ~10 个的敏感能力（Bash/WebFetch）
  是样本**如实声明**的，一致性核验按设计判 allow。**这正是"仅靠能力粒度"的固有盲区**，
  入口触发解决不了它——需要下一节的"可疑目标信誉"信号。
- 良性侧：入口触发**未引入任何新误报**（FPR 仍 0.00）——良性入口即便被调用，行为要么无敏感性、
  要么已在声明范围内。

**结论**：入口触发是**正确且必要**的鲁棒性改进（对"函数门控型"良构恶意样本确实能抓到，已由单测锁定），
但它**没有、也不应被宣称**在本批提升了召回。要在真实数据上把召回撬起来，真正的两个杠杆是：
1. **可疑外传目标信誉**：即便网络能力"已声明"，外联到 discord CDN / telegram / rentry / pastebin raw /
   硬编码 IP 等**信誉可疑目标**仍应作为强信号（可打掉上面那 ~10 个"声明了广权限但外传到恶意目标"的样本，
   且良性几乎不会碰这些目标，预期不牺牲 FPR）。
2. **样本可执行性**：为 Windows 定向 / 缺库 / 语法非法的样本提供更贴近其预期运行环境的执行路径
   （或明确将这部分划归静态/LLM 路径，不计入运行时召回分母）。

## 6. 复现方式

```bash
source .venv/bin/activate
python -m scripts.run_pilot_batch --n 50 --timeout 20 --out runs/pilot50.json
```

> 注意：会在隔离 Docker 内执行真实恶意代码，请确保 Docker 可用且以隔离方式运行。
